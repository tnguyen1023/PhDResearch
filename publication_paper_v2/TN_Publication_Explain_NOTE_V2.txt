======

Persona Layer — P, GK, τᵈ, τᵈᵖ, τᵈ⁰, Δ, Δₚ, q, ρₘₐˣ, H
These ten parameters are the newest and most distinctive part of the formulation. Everything before this — the network inventory, the attack picture, the constraints, the value adjustments — existed in some form in prior honeypot placement work. The persona layer is what makes this formulation different.
The central insight is simple: attackers do not just identify honeypot types, they map identities. A patient attacker does not think "there is an SSH server at 10.2.3.45." They think "Alice Smith's HR workstation has been at 10.2.3.45 for three months — I know its hostname, its AD account, its SSL certificate, its login schedule, and its network neighbors." Changing the SSH service does nothing against this attacker. Only changing the identity — the complete package of hostname, account, certificate, and behavior — forces them to restart their reconnaissance from scratch.
The persona layer gives the solver the parameters it needs to plan, execute, and continuously adapt an identity-rotation schedule across a multi-month horizon.

                      Hostname: findb-prod-01.firstnational.local

GKᵢ,ₚ — Generative Knowledge Set (the plausibility check)
GK is the set of (server type, persona) pairs that are believable in a real enterprise environment. For each honeypot type i and persona p, GK_{i,p} contains the set of (servertype, persona) combinations that a knowledgeable attacker would not immediately reject as implausible.

——

H — Planning Horizon (total number of deployment slots)
H is the scalar count of months the solver is planning for. T̄ = {1, …, H} is the set of all planning slots. t ∈ T̄ is a specific month in the schedule.

——

τᵈ — Type Discovery Threshold (how long before the service behavior gives it away)

τᵈ = 3 months

Month 1: ssh_trap, DMZ, HR_workstation → u_{ssh,DMZ,1} = 0  (credit earned)
Month 2: ssh_trap, DMZ, HR_workstation → u_{ssh,DMZ,2} = 0  (credit earned)
  [Persona rotated to DevOps_server — τᵈᵖ threshold hit]
Month 3: ssh_trap, DMZ, DevOps_server  → u_{ssh,DMZ,3} = 0  (credit earned)
  [Type has now been active for 3 consecutive months]
Month 4: ssh_trap, DMZ, Finance_DB    → u_{ssh,DMZ,4} = 1  ← TYPE BURNED

From month 4 onward: zero credit regardless of persona.
The attacker has identified the SSH implementation as fake.
Changing to Finance_DB persona changes nothing — the
behavioral signature of the SSH service is already in
their notes.


——

τᵈᵖ — Persona Discovery Threshold (how long before the identity is mapped)
τᵈᵖ is the number of consecutive months a specific persona can remain visible in the same zone before an attacker builds a complete dossier on that identity and it becomes useless as a deception.

τᵈᵖ = 2 months

Month 1: ssh_trap, Internal, Finance_DB
  u_{ssh,Internal,1,Finance_DB} = 0  (persona clock starts)
  Attacker begins mapping "findb-prod-01.firstnational.local"

Month 2: ssh_trap, Internal, Finance_DB
  u_{ssh,Internal,2,Finance_DB} = 0  (clock at 1 — threshold τᵈᵖ=2 not yet hit)
  Attacker completes their dossier on findb-prod-01

Month 3: MUST ROTATE — if Finance_DB ran again:
  u_{ssh,Internal,3,Finance_DB} = 1  ← PERSONA BURNED

  Instead, solver rotates to DevOps_server:
  u_{ssh,Internal,3,Finance_DB}   = 0  (Finance_DB clock resets — gone)
  u_{ssh,Internal,3,DevOps_server} = 0  (new clock starts)

  Attacker's dossier on findb-prod-01 is now worthless.
  devbuild-02.firstnational.local appeared in its place.
  Attacker must start over.

——


The independence of τᵈᵖ from τᵈ:
These two clocks run independently and reset independently. This independence is the key architectural insight.

Type clock (τᵈ): counts consecutive months the SSH SERVICE is active
  Resets only when: the honeypot turns off for at least one month
  Persona changes do NOT reset it

Persona clock (τᵈᵖ): counts consecutive months a SPECIFIC IDENTITY is visible
  Resets when: the persona changes to any different persona
  The type clock keeps running regardless


This means persona rotation is cheap — it resets τᵈᵖ without affecting τᵈ, costs one Δₚ churn slot, and requires only a configuration change (new hostname, new AD account, new certificate). Type rotation is expensive — it is the only way to reset τᵈ, costs one Δ churn slot, requires taking the honeypot offline and reconfiguring the service implementation, and leaves a coverage gap while the honeypot is down.
The solver exploits this asymmetry. It rotates personas frequently (cheap, effective against identity mapping) and rotates types only when necessary (expensive, required only when behavioral fingerprinting becomes the threat).

———


τᵈ⁰ — Threat-Adaptive Rotation Baseline (the rotation speed at zero threat)
τᵈ⁰ is the value of τᵈ when the attack probability on a path is exactly zero. It is the starting point for the threat-adaptive rotation formula and represents the slowest, most operationally relaxed rotation schedule the solver will ever produce.
The formula that uses τᵈ⁰:


τᵈ(i,z,t) = τᵈ⁰ × (1 − ρπ(t) / ρₘₐˣ)
When ρπ = 0 (no threat on this path), τᵈ = τᵈ⁰. The honeypot can stay active for τᵈ⁰ months before rotation. When ρπ rises toward ρₘₐˣ, τᵈ shrinks toward 0, forcing increasingly rapid rotation.
τᵈ⁰ = 3 months   (baseline rotation speed)
ρₘₐˣ = 1.0       (ceiling for the formula)

At ρπ = 0.00 (no threat):
  τᵈ = 3 × (1 − 0.00/1.0) = 3.00 → rotate every 3 months
  Maximum operational comfort. Team reconfigures quarterly.

At ρπ = 0.35 (normal threat for web-to-database path):
  τᵈ = 3 × (1 − 0.35/1.0) = 1.95 → rotate approximately every 2 months
  Normal operating tempo. Rotation is bimonthly.

At ρπ = 0.52 (elevated threat after FIN-Payroll alert):
  τᵈ = 3 × (1 − 0.52/1.0) = 1.44 → rotate approximately every 1.5 months
  Heightened posture. Rotation is between monthly and bimonthly.

At ρπ = 0.75 (high threat, active campaign detected):
  τᵈ = 3 × (1 − 0.75/1.0) = 0.75 → rotate approximately every month
  High alert. Monthly rotation mandatory.

At ρπ = 0.90 (severe threat, confirmed targeting of this bank):
  τᵈ = 3 × (1 − 0.90/1.0) = 0.30 → rotate approximately every 9 days
  Emergency posture. Nearly weekly rotation.

———

Why τᵈ⁰ is a separate parameter from τᵈ:
τᵈ changes every time ρπ changes. τᵈ⁰ is the stable anchor that does not change — it is the policy decision about how aggressively to rotate at baseline. A risk-averse organization might set τᵈ⁰ = 2 (rotate every 2 months even at zero threat). A resource-constrained organization might set τᵈ⁰ = 6 (rotate only every 6 months at baseline). The formula then scales from this anchor based on live threat intelligence.

Δ — Type Rotation Churn Limit (maximum type on/off switches)
Δ is the maximum number of times a honeypot type can switch between active and inactive in a given zone across the entire planning horizon H. It is a hard cap on operational reconfiguration work.
Why Δ exists — the operational reality:
Every time a honeypot type turns off and back on, the operations team must:
	1	Decommission the current honeypot instance (archive logs, update asset inventory, notify monitoring team)
	2	Reconfigure the honeypot software (new implementation to reset the behavioral fingerprint)
	3	Recommission the honeypot instance (install, configure, test, register in monitoring)
	4	Verify the new instance is correctly logging (smoke test, monitoring confirmation)
This takes approximately half a day of engineer time per reconfiguration. With a team of three security engineers who also manage the SIEM, firewall rules, incident response, and compliance documentation, more than two reconfigurations per zone per planning horizon is genuinely unsustainable.
Real example at First National Financial:


Δ = 2 for the ssh_trap in the DMZ zone

This means: across the entire 4-month planning horizon,
the ssh_trap in DMZ can switch on→off or off→on at most twice.

Valid schedules (each state change marked with ↕):
  1,1,0↕,1↕  → 2 changes (off at t=3, on at t=4) ✓
  1,0↕,1↕,1  → 2 changes (off at t=2, on at t=3) ✓
  1,0↕,0,0   → 1 change  (off at t=2, stays off) ✓
  1,1,1,1     → 0 changes (always on) ✓

Invalid schedules:
  1,0↕,1↕,0↕ → 3 changes — EXCEEDS Δ = 2 ✗
  0,1↕,0↕,1↕ → 3 changes — EXCEEDS Δ = 2 ✗
How Δ interacts with τᵈ:
τᵈ says "you must rotate after 3 months." Δ says "you can only rotate twice." Together they create a bounded rotation schedule. With τᵈ = 3, H = 4, and Δ = 2:


The solver knows:
- The ssh_trap cannot run for 4 consecutive months (τᵈ = 3 → burned in month 4)
- It can turn off and back on at most twice (Δ = 2)
- It must cover at least 2 months on the web-to-database path (C10, ρ=0.35, H=4)

Valid optimal schedule:
  Month 1: ON  (type clock = 1)
  Month 2: ON  (type clock = 2)
  Month 3: ON  (type clock = 3 — approaching threshold)
  Month 4: OFF ↕ (type clock resets — 1 change used)

This satisfies: τᵈ never hit (off before month 4 would burn it)
                Δ satisfied (1 change used, max 2)
                C10 satisfied (3 months covered > 2 required)

Δₚ — Persona Churn Limit (maximum identity changes)
Δₚ is the maximum number of times a honeypot's persona can change in a given zone across the entire planning horizon H. It bounds how frequently the operations team must create new fake identities, update AD accounts, rotate certificates, and reconfigure traffic patterns.
Why Δₚ exists and why it is separate from Δ:
Persona changes are cheaper than type rotations — no software reinstallation, no monitoring gap, no full reconfiguration. But they still cost engineer time: create the new AD account, issue the new certificate, configure the new hostname and DNS record, update the traffic simulation pattern, test the new identity for plausibility, and notify the monitoring team.
Each persona change takes approximately two to three hours. With Δₚ = 2 and H = 4 months, the team handles at most two persona changes per honeypot per zone across the entire quarter — roughly one change every other month on the most active honeypots. This is manageable.
Real example:


Δₚ = 2 for ssh_trap in Internal LAN zone

Across H = 4 months, the ssh_trap can change its
persona identity at most twice.

Valid rotation schedules:
  HR → HR → DevOps↕ → DevOps  (1 change) ✓
  HR → HR → DevOps↕ → Finance↕ (2 changes) ✓
  HR → DevOps↕ → DevOps → Finance↕ (2 changes) ✓

Invalid:
  HR → DevOps↕ → Finance↕ → Linux↕  (3 changes) — EXCEEDS Δₚ ✗
Δₚ vs Δ in the solver's decision-making:


The solver wants to keep the ssh_trap in Internal
continuously active (to maximize coverage on three
overlapping attack paths) while avoiding both
persona discovery (τᵈᵖ = 2) and type discovery (τᵈ = 3).

With Δₚ = 2 and H = 4:
  Month 1: ssh_trap, Internal, HR_workstation
  Month 2: ssh_trap, Internal, HR_workstation    ← τᵈᵖ clock = 1
  Month 3: ssh_trap, Internal, DevOps_server ↕  ← persona change 1, τᵈᵖ(HR) resets
  Month 4: ssh_trap, Internal, DevOps_server     ← τᵈᵖ clock for DevOps = 1

Result: 2 persona changes used (= Δₚ exactly)
        Type never turned off (no Δ churn used)
        No persona ever reaches τᵈᵖ = 2
        All four months covered for path credit

This is the solver finding the optimal rotation schedule
that satisfies all constraints simultaneously.

qₚ — Persona Prior (the LLM's assessment of which identities to prioritize)
qₚ ∈ [0,1] is the probability weight the solver assigns to each persona p when calculating detection credit. All priors must sum to 1 across all personas in P. It is updated by Algorithm 1 whenever threat intelligence changes.
What qₚ means in the credit formula:
Every soft-clause credit is multiplied by qₚ. A Finance_DB honeypot deployment that would normally earn W = 1.995 credit units earns 1.995 × q_{Finance_DB} instead. If q_{Finance_DB} = 0.45, the deployment earns 1.995 × 0.45 = 0.898. If q_{Finance_DB} = 0.10, the same deployment earns only 1.995 × 0.10 = 0.200.
Real example — qₚ in normal conditions:


No specific threat intelligence. Uniform priors.

q_{Finance_DB}    = 0.25
q_{HR_workstation}= 0.25
q_{DevOps_server} = 0.25
q_{Generic_Linux} = 0.25

All personas equally prioritized. The solver chooses
based purely on network topology, path coverage, and
technique diversity. No attacker-motivation weighting.
Real example — qₚ after the FIN-Payroll alert:


February 15th: FS-ISAC alert about ransomware group
targeting financial sector payroll databases.

Algorithm 1 reads the alert, classifies the attacker
as "financially motivated, targeting payroll systems,"
and adjusts priors:

q_{Finance_DB}    = 0.45  (attacker is hunting payroll data)
q_{HR_workstation}= 0.30  (HR machines often hold supplementary
                            employee data — secondary target)
q_{DevOps_server} = 0.15  (irrelevant to financial data theft)
q_{Generic_Linux} = 0.10  (irrelevant to this campaign)

Effect on credit:
  Finance_DB deployment now earns 0.45/0.25 = 1.8× more
  than at default priors.

  DevOps_server deployment now earns 0.15/0.25 = 0.6×
  as much — the solver deprioritizes it.

The solver re-runs and concentrates additional Finance_DB
coverage where it can fit within constraints.
Real example — qₚ after an IP-theft indicator:


A different alert: nation-state actor targeting
financial firms for source code and internal tooling.

q_{Finance_DB}    = 0.10  (financial data not the target)
q_{HR_workstation}= 0.15  (some interest in employee credentials)
q_{DevOps_server} = 0.45  (source code repositories, CI/CD — primary target)
q_{Generic_Linux} = 0.30  (infrastructure reconnaissance)

Effect: the solver now prioritizes DevOps_server personas
across the network. Finance_DB honeypots are deprioritized
unless they sit on critical paths with no alternative.
Why qₚ is a stronger LLM contribution than adjusting anonymous weights:
When Algorithm 1 outputs updated qₚ values, the security team receives an operationally interpretable directive: "This month, prioritize Finance_DB and HR_workstation personas." A security analyst can read that, understand it, debate it, and override it if domain knowledge warrants.
If Algorithm 1 instead output updated wⱼ values — "increase w_{T1048} from 0.90 to 0.95" — the analyst has no way to evaluate whether that adjustment is correct without understanding the full weight formula and its interactions. The qₚ output is transparent. The wⱼ output is opaque.

ρₘₐˣ — Maximum Path Probability (the rotation formula ceiling)
ρₘₐˣ is a technical parameter that prevents the threat-adaptive rotation formula from producing impossible or degenerate results. It is the maximum path probability value used in the denominator of the rotation formula.
The formula it anchors:


τᵈ(i,z,t) = τᵈ⁰ × (1 − ρπ(t) / ρₘₐˣ)
What goes wrong without ρₘₐˣ:
If ρπ could equal 1.0 in the formula and ρₘₐˣ = 1.0:


τᵈ = τᵈ⁰ × (1 − 1.0/1.0) = τᵈ⁰ × 0 = 0
A τᵈ of zero means the honeypot must rotate every zero months — continuously, infinitely. This is physically impossible and operationally meaningless. The formula has collapsed.
ρₘₐˣ prevents this:


ρₘₐˣ = 1.0 (typical setting)

If threat intelligence reports ρπ = 0.95 (very high):
  τᵈ = 3 × (1 − 0.95/1.0) = 3 × 0.05 = 0.15 months ≈ 5 days

This is aggressive but finite. The solver interprets this as:
"rotate approximately every week." Operationally demanding
but not impossible.

If ρπ somehow reaches exactly 1.0:
  τᵈ = 3 × (1 − 1.0/1.0) = 0

At this point ρₘₐˣ serves as a clamp. The solver enforces
τᵈ ≥ 1 as a hard minimum — you must always have at least
one slot between rotations, because zero-slot rotation is
impossible.
A more conservative ρₘₐˣ setting:
Some organizations set ρₘₐˣ = 0.80, which ensures τᵈ never drops below τᵈ⁰ × (1 − 1.0/0.8). This prevents the formula from producing degenerate results even when threat intelligence reports unrealistically high probabilities.

Together these ten parameters give the solver everything it needs to answer the hardest question in honeypot placement: not just where to put honeypots, and not just what they should be — but what they should pretend to be, for how long, and how to change their identities continuously so that a patient attacker's reconnaissance never accumulates into actionable intelligence.

T = {
  T1190,  — exploit public-facing application (web attack)
  T1021,  — remote services via SSH (lateral movement)
  T1110,  — brute force login attempts
  T1213,  — collecting data from information repositories
  T1048,  — exfiltration over alternative protocol (disguised as HTTPS)
  T1558,  — steal Kerberos tickets (credential attack on Active Directory)
  T1003,  — credential dumping
  T0855,  — modify system settings (ICS/SCADA attack)
}


—— Each honeypot type covers a fixed subset of T:

web_trap covers:   T1190, T1133, T1059
ssh_trap covers:   T1021, T1110, T1078
db_trap covers:    T1213, T1048, T1485
ad_trap covers:    T1558, T1003, T1098
scada_trap covers: T0855, T0814, T0869



Why it matters: An attacker who knows that certain techniques go undetected will specifically use those techniques. If T1048 is not covered by any honeypot, an attacker can exfiltrate data disguised as HTTPS traffic and walk out undetected. T tells the solver which attack behaviors remain in the blind spot so it can prioritize covering them.
T is also where ATT&CK differs from attack paths (G). G describes the route through the network — which zone the attacker enters, which zone they pivot through, which zone their target is in. T describes what specific action they take at each step along that route. You need both: the route tells you where to put honeypots, the technique tells you which honeypot type can actually catch the attacker at that location.

A = {
  — DMZ zone (3 servers):
    web01, web02, web03         — public banking web servers

  — Internal LAN zone (50 servers):
    dc01, dc02                  — Active Directory domain controllers
    filesvr01..filesvr08        — Windows file servers
    oracle-prod                 — production Oracle database (transaction records)
    oracle-dev                  — development database
    ws01..ws40                  — employee workstations

  — Cloud zone (12 servers):
    app-server-01..app-server-08 — application servers
    s3-gateway                  — S3 bucket gateway
    api-gw                      — API gateway

  — OT zone (4 devices):
    hvac-ctrl-01, hvac-ctrl-02  — HVAC controllers
    bms-main                    — building management system

  — Management zone (8 devices):
    fw-mgmt                     — firewall management console
    sw-mgmt-01..sw-mgmt-06      — network switch management
    siem-server                 — SIEM logging server
}

Each asset has two properties that feed into the value calculation:
dₘ,ₐ — how critical the asset is:
	•	oracle-prod: dₘ = 3.5 (10 years of financial transaction records — catastrophic if compromised)
	•	dc01, dc02: dₘ = 3.0 (Active Directory compromise means total network takeover)
	•	filesvr01..08: dₘ = 1.5 (important but recoverable with backups)
	•	ws01..ws40: dₘ = 1.0 (significant but not catastrophic individually)
	•	hvac-ctrl: dₘ = 0.8 (disruption is uncomfortable but not business-critical)

hᵈ,ₐ — how many hops from the network entry point:
	•	web01, web02, web03: hᵈ = 1 (directly internet-facing, first hop)
	•	Internal servers: hᵈ = 2 (attacker must traverse the DMZ first)
	•	Deep backend oracle-prod: hᵈ = 3 (attacker must cross DMZ, then Internal, then reach DB subnet)
The weight formula combines these: a honeypot protecting oracle-prod at hop 1 earns 3.5 / 1 = 3.5 units of weight. The same honeypot at hop 3 earns only 3.5 / 3 = 1.17 units. This is the mathematical encoding of a simple security truth — stopping an attacker before they reach the target is worth far more than catching them after they arrive.
Why it matters: Without A, the solver cannot calculate what catching an attacker is actually worth. It would treat all detection events as equally valuable. A makes the solver prefer zones with high-value, close-to-entry assets — which is exactly where experienced security teams put their best sensors.

Z = {
  DMZ,       — internet-facing, highest attack exposure
  Internal,  — office network, lateral movement hub
  Cloud,     — AWS-hosted, mixed trust
  OT,        — industrial systems, physically isolated
  Management — network infrastructure, highest privilege
}

Why the zones are structural, not just labels:
Zone boundaries enforce three real constraints on what the solver can do:
First, air gaps. The OT zone is physically isolated from the DMZ and Cloud zones. There is no network cable connecting them. The solver cannot place a web_trap in the OT zone because the web_trap needs to receive HTTP connections from the internet, and the internet cannot reach OT. No matter how valuable such a deployment would theoretically be, the physical wire does not exist.
Second, eligibility. The scada_trap speaks industrial protocols — Modbus, DNP3, OPC-UA. These protocols only exist in the OT zone. Deploying a scada_trap in the Cloud zone would be like putting a deep-sea diving suit in a shopping mall — nobody there has any reason to interact with it.
Third, per-zone budgets. The OT team and the network security team have separate budget lines. B_{OT} = 2,000 units per month. Even if the global budget B has unused room, the solver cannot spend more than 2,000 in OT. A second scada_trap (cost 2.0) is blocked not because the global budget is exhausted but because the zone sub-limit is fully consumed.
How zones shape the solver's decisions:
The five zones create a structured search space. The solver does not evaluate every possible host — it evaluates every possible (honeypot type, zone) pair. An 8-type × 5-zone space gives 40 combinations per time slot. Most of these are immediately eliminated by eligibility rules (scada_trap only in OT), air-gap constraints (no web_trap in OT), and budget limits (no second scada_trap when B_{OT} is spent). What remains is a small, well-constrained problem that RC2 can solve to certified optimality in seconds.

How K, T, A, Z fit together
These four sets define the skeleton of the problem before a single deployment decision is made:


K tells the solver: here are your tools
T tells the solver: here is what each tool can catch
A tells the solver: here is what you are protecting and how much each asset is worth
Z tells the solver: here is the terrain and where each tool is allowed to operate

Together they constrain the search space from the theoretical maximum — any honeypot, anywhere, detecting anything — down to a small set of feasible, meaningful deployments that fit the actual network and actual budget. Everything else in the formulation (attack paths, constraints, persona layer) operates on top of this skeleton.

=====

Attack Picture — G, ρ, iv
These three parameters answer one question: what does the solver know about the enemy before it starts planning?
Without G, ρ, and iv, the solver would treat all honeypot placements as equally valuable. An SSH trap in a quiet management subnet would be worth the same as an SSH trap sitting directly on the path an attacker takes to reach the database. These three parameters are what make the solver care about where attackers go, how often they go there, and how much it costs the defender to miss them at each step.

G = (π₁, …, πᴺ) — Attack Paths
G is the set of routes the security team believes attackers will take through the network. Each path π is an ordered sequence of zone hops — a script describing the journey from initial entry to final target.

π₁ — Web breach to database (most common)
     Internet → DMZ → Internal LAN → Oracle DB
     An attacker exploits the banking web portal,
     pivots into the internal network, and targets
     the transaction database.

π₂ — Credential stuffing to Active Directory
     Internet → Cloud portal → Internal LAN → Domain Controller
     An attacker brute-forces cloud credentials,
     uses them to reach the internal network, and
     escalates to full domain control via AD.

π₃ — Supply chain via cloud entry
     Internet → Cloud → Cloud → Internal LAN
     An attacker compromises a cloud-hosted
     application, moves laterally within cloud,
     then pivots into the internal network.

π₄ — Insider threat through building systems
     OT zone → Internal LAN → File servers
     A malicious insider or physical intruder
     enters through building management systems
     and moves toward sensitive file storage.

π₅ — Direct brute force to management
     Internet → DMZ → Management zone
     An attacker hammers SSH on exposed
     management interfaces to gain privileged
     network access.

What each path contains:
Each path π has three things:
	•	A list of zones in order, from entry to target
	•	A probability ρπ (how likely this route is)
	•	An interception value ivπ,h at each hop h (how much it is worth to catch the attacker there)
Why ordered sequences matter:
The order is everything. π₁ goes DMZ → Internal → DB. This means:
	•	A honeypot in the DMZ catches the attacker at hop 1, before they have touched anything else
	•	A honeypot in Internal catches them at hop 2, after they have already entered the building but before they reach the database
	•	A honeypot at the database catches them at hop 3, but by then they are already at the target
Catching at hop 1 prevents all subsequent damage. Catching at hop 3 only records what has already happened. The path ordering is what gives the formulation its prevention-versus-forensics distinction. Without ordered paths, every detection event would look the same regardless of when in the intrusion it occurred.
Where G comes from in practice:
First National Financial already runs RedSeal Network Security Manager, which continuously maps the network topology and generates attack path reports. Every week the dashboard shows the ten most probable attacker routes based on current firewall rules and known vulnerabilities. The security team exports these in STIX format. Algorithm 1 ingests this feed and updates ρ values when new threat intelligence arrives.

=====

ρπ — Path Probability (how likely each path is)
ρπ is a number between 0 and 1 expressing the security team's current belief about how likely an attacker is to follow path π rather than the other paths.

ρ_{π₁} = 0.35  — web breach to database
           Most likely. Web-facing systems are the highest-volume
           attack surface. Financial sector sees daily web application
           attacks. 35 out of 100 attacks follow this route.

ρ_{π₂} = 0.25  — credential stuffing to AD
           Second most likely. Credential attacks are extremely
           common. Active Directory is the crown jewel for
           attackers seeking persistence.

ρ_{π₃} = 0.20  — cloud supply chain
           Growing threat as cloud adoption increases.
           Attackers increasingly use cloud as entry point.

ρ_{π₄} = 0.10  — insider via OT
           Lower probability but high impact. Requires
           physical access or insider knowledge.

ρ_{π₅} = 0.10  — brute force to management
           Lower probability because management interfaces
           are usually behind VPNs and additional controls.

Total: 0.35 + 0.25 + 0.20 + 0.10 + 0.10 = 1.00

How ρ shapes what the solver values:
Every detection credit is multiplied by ρπ. A honeypot covering a hop on π₁ (ρ=0.35) earns 3.5× more credit than a honeypot covering the same hop on π₄ (ρ=0.10), all else equal. The solver naturally concentrates resources on high-probability paths.
How ρ changes in real time:
On February 15th, FS-ISAC (Financial Services Information Sharing and Analysis Center) releases an alert: the ransomware group "FIN-Payroll" has been actively scanning financial sector web applications and specifically targeting Oracle databases. The STIX feed updates:


Before alert: ρ_{π₁} = 0.35
After alert:  ρ_{π₁} = 0.52
Algorithm 1 fires automatically. It recomputes all the soft-clause weights that depend on ρ_{π₁}, warm-starts RC2 from the current deployment, and produces a revised schedule in under 5 seconds. The honeypots covering π₁ earn more credit now, so the solver adds more coverage on that path in the remaining months of the planning horizon. The security team follows the new schedule without manually reconfiguring anything.
The connection to C10:
ρπ also feeds directly into the path-persistence constraint C10, which requires that path π is covered in at least ⌈ρπ × H⌉ slots across the planning horizon. When ρ_{π₁} rises from 0.35 to 0.52, the minimum required coverage slots jump from ⌈0.35 × 4⌉ = 2 months to ⌈0.52 × 4⌉ = 3 months. The solver must ensure no month is left uncovered on the web-to-database path. A patient attacker who knew the old schedule planned for 2 covered months can no longer wait for a gap — there is now no gap to wait for.

=====

ivπ,h — Intercept Value at Each Hop (how much each catch is worth)

ivπ,h is the value of catching the attacker at hop h of path π. It encodes the prevention-versus-forensics distinction in a single number per hop.

π₁: Internet → DMZ → Internal LAN → Oracle DB

iv_{π₁, hop1} = 2.0   — DMZ (entry point)
  Catching the attacker here stops everything.
  They have not touched any internal system.
  No data accessed. No lateral movement.
  Full prevention value.

iv_{π₁, hop2} = 1.4   — Internal LAN (halfway through)
  The attacker has entered the building but not
  reached the database. Some reconnaissance has
  occurred. Catching here stops the database breach
  but does not undo the perimeter compromise.
  Partial prevention value.

iv_{π₁, hop3} = 0.8   — Oracle DB (at the target)
  The attacker has reached the database. Catching
  them here is forensic — it logs what they did
  and who they are, but the data has already been
  accessible. Response and attribution value only.
  Forensic value, not prevention.


The math that follows:
The path weight PW combines all three parameters:

PW_{π₁, hop1, T1190, web_server} = ρ_{π₁} × iv_{π₁,1} × W_{T1190, web_server}
                                  = 0.35   × 2.0        × W_{T1190, web_server}
                                  = 0.70   × W_{T1190, web_server}

PW_{π₁, hop3, T1048, oracle_db}  = ρ_{π₁} × iv_{π₁,3} × W_{T1048, oracle_db}
                                  = 0.35   × 0.8        × W_{T1048, oracle_db}
                                  = 0.28   × W_{T1048, oracle_db}


A web_trap catching T1190 at hop 1 earns 0.70 × W per credit. A db_trap catching T1048 at hop 3 earns only 0.28 × W per credit — less than half — even if the db_trap costs the same and covers an equally stealthy technique. The hop position alone creates a 2.5× difference in value.
Why interception values decrease as hops increase:
This is not arbitrary. It reflects a genuine security principle: the further along the path the attacker has traveled, the less catching them prevents. At hop 1 you prevent the entire intrusion. At hop 2 you prevent the final target compromise but not the initial access. At hop 3 you record the breach but do not prevent it.
The L4 soft clause in the objective reinforces this with a 1000× weight on early interception. The first non-final-hop coverage of any path earns the prevention bonus — the single highest-weighted event in the entire formulation. This forces the solver to always maintain at least one prevention-layer honeypot on every path regardless of what other credit is available.
Different paths have different iv profiles:


π₄: OT zone → Internal LAN → File servers (2 hops)
  iv_{π₄, hop1} = 1.6   — OT entry (rare but high-impact to catch early)
  iv_{π₄, hop2} = 0.9   — File servers (attacker already inside)

π₅: Internet → DMZ → Management zone (2 hops)
  iv_{π₅, hop1} = 1.8   — DMZ (management attacks are high-privilege)
  iv_{π₅, hop2} = 1.1   — Management zone (catching here still prevents total network compromise)
π₅ has higher interception values per hop than π₁ because a management zone compromise gives the attacker control over every network device — the damage potential is extreme even at the final hop.

How G, ρ, and iv fit together: a complete calculation
Here is how the three parameters combine for a single deployment decision:
Decision: Deploy an ssh_trap in the Internal LAN zone during month 2, wearing DevOps_server persona.
The Internal LAN is hop 2 of π₁, hop 2 of π₂, and hop 2 of π₄. This single deployment covers three paths simultaneously — the cross-path force-multiplier at work.

Credit from π₁ (web to database):
  ρ_{π₁} × iv_{π₁,2} × W_{T1021} × q_{DevOps}
= 0.35   × 1.4        × W_{T1021} × 0.20
= 0.098 × W_{T1021}

Credit from π₂ (credential stuffing to AD):
  ρ_{π₂} × iv_{π₂,2} × W_{T1021} × q_{DevOps}
= 0.25   × 1.5        × W_{T1021} × 0.20
= 0.075 × W_{T1021}

Credit from π₄ (insider via OT):
  ρ_{π₄} × iv_{π₄,2} × W_{T1021} × q_{DevOps}
= 0.10   × 0.9        × W_{T1021} × 0.20
= 0.018 × W_{T1021}

Total credit from one deployment:
  (0.098 + 0.075 + 0.018) × W_{T1021} = 0.191 × W_{T1021}


=====

Constraints — C, I₂, ⋄, cost, B, B₂
These five parameters define the hard reality the solver must work within. They are not preferences or soft guidelines — they are inviolable rules. The solver cannot trade them off against objectives. No amount of detection credit justifies violating any of them.
Think of them as the legal and physical boundaries of the problem. The objectives tell the solver what to want. The constraints tell it what it is allowed to do.

C ⊆ K×K — Pairwise Conflict Set (honeypots that cannot coexist)
C is the set of honeypot type pairs that cannot run simultaneously in the same time slot. Each pair (kᵢ, kₗ) in C means: if you deploy type kᵢ, you cannot also deploy type kₗ that month, and vice versa.
Why conflicts exist — three real reasons:
Reason 1 — Shared monitoring pipeline. Two honeypots that capture the same attacker behavior flood the same logging system with duplicate alerts. The monitoring team cannot tell the signals apart. What looks like doubled coverage is actually doubled noise with no additional detection value.
Reason 2 — Shared physical infrastructure. Some honeypot types require the same network tap, the same monitoring server, or the same logging database. Running both simultaneously overloads the shared resource and degrades the quality of both.
Reason 3 — Contradictory network signals. Two honeypots presenting conflicting identities on the same network segment can confuse each other's deception. An attacker scanning the network sees inconsistent behavior and correctly infers that something artificial is present.


C = {
  (generic_trap, dns_trap),    — both capture T1046 network scanning;
                                 running both saturates the same SIEM
                                 alert queue without additional coverage

  (web_trap, generic_trap),    — both expose HTTP services; generic_trap
                                 mimics web behavior and creates ambiguous
                                 signals when web_trap is also present

  (smb_trap, generic_trap),    — both capture T1021 SMB lateral movement;
                                 duplicate signals confuse incident response
}

How C works in the constraint:
The constraint C4 says: for every conflicting pair (kᵢ, kₗ) in C, at most one of them can be active in any given time slot t, across any zone and any persona.

Month 1 scenario: The solver wants to deploy generic_trap in Internal
                  AND dns_trap in DMZ simultaneously.

C4 blocks this: (generic_trap, dns_trap) ∈ C
                x_{generic, t=1} = 1 forces x_{dns, t=1} = 0
                The dns_trap cannot run in month 1 at all.

The solver must choose one. It calculates which earns more credit — dns_trap covering T1572 Protocol Tunneling (σ=0.85, rare and stealthy) versus generic_trap covering T1046 Network Scanning (σ=0.20, common and already caught by other means) — and picks the dns_trap. The generic_trap stays off that month.
Why this matters for the budget result:
At XLarge scale, the solver uses only approximately 0.1% of the global budget B. This is not a modeling error. It is a direct consequence of C. Conflict constraints prevent the solver from stacking up redundant honeypots. After deploying the four most complementary types (ssh_trap, db_trap, dns_trap, ad_trap), adding a fifth type either conflicts with one of the four or produces redundant technique coverage. The solver correctly recognizes that the fifth deployment earns less credit than it costs and leaves the remaining budget unspent. C is what makes the portfolio sparse and targeted rather than maximal.

Value Adjustments — w, dₘ, hᵈ, σ
These four parameters answer one question: how much is each honeypot deployment actually worth?
The network inventory (K, T, A, Z) told the solver what exists. The attack picture (G, ρ, iv) told it where attackers go. The constraints (C, I₂, ⋄, cost, B, B₂) told it what is allowed. Now these four parameters tell it how to put a precise value on each possible deployment — how to calculate the price tag on detection credit before the solver starts comparing options.

=====

hᵈ,ₐ — Hop Distance from Entry (perimeter preference)
hᵈ,ₐ is the number of network hops an attacker must traverse to reach asset a from the network entry point. It encodes the most important security principle in the formulation: catching an attacker early in their journey is worth far more than catching them late.

Asset                    Zone        hᵈ      Meaning
─────────────────────────────────────────────────────────────────────

                                             an attacker from outside touches.

The weight formula that uses hᵈ:

w̃_{j,a} = w_j × dₘ,a / hᵈ,a        (equation 2)

This single formula encodes a profound operational insight. Let us compute it for three specific scenarios:


Scenario A: ssh_trap near oracle-prod at hop 3
  w̃ = 0.75 × 3.5 / 3 = 0.875

Scenario B: ssh_trap near a workstation at hop 2
  w̃ = 0.75 × 1.0 / 2 = 0.375

Scenario C: web_trap near web01 at hop 1
  w̃ = 0.55 × 1.5 / 1 = 0.825


Scenario A covers the most critical asset (oracle-prod, dₘ=3.5) but at hop 3. Scenario C covers a less critical asset (web server, dₘ=1.5) but at hop 1. They end up with similar values (0.875 vs 0.825) because the perimeter advantage of hop 1 nearly offsets the lower asset criticality. The formula automatically balances criticality against proximity to entry.
Why dividing by hᵈ is the right choice:
Imagine the web-to-database attack path π₁:


Hop 1 — DMZ web server:
  Attacker has entered through the web portal.
  No internal data accessed yet.
  Catching here = full prevention.
  Damage prevented: 100% of the potential breach.

Hop 2 — Internal LAN:
  Attacker has crossed the DMZ and is moving through
  the internal network. They have conducted reconnaissance.
  Catching here = prevent database breach but not
  the initial perimeter compromise.
  Damage prevented: roughly 70% of the potential breach.

Hop 3 — Oracle database:
  Attacker has reached the database. They may have
  already run queries. Catching here = forensic evidence
  but the database was accessible.
  Damage prevented: roughly 30% — attribution only.


Dividing by hᵈ makes the value at hop 1 three times the value at hop 3, which roughly matches the actual damage-prevention ratio. The formula is not perfectly calibrated — no formula could be — but it is in the right direction and the right order of magnitude.
The combined effect of dₘ and hᵈ — a full worked example:


The solver is deciding between two placements:

Option 1: db_trap near oracle-prod in the DB subnet
  Asset: oracle-prod, dₘ = 3.5, hᵈ = 3
  Technique: T1213 data collection, w = 0.80
  w̃ = 0.80 × 3.5 / 3 = 0.933

Option 2: ssh_trap near the DMZ web server
  Asset: web01, dₘ = 1.5, hᵈ = 1
  Technique: T1021 lateral movement, w = 0.75
  w̃ = 0.75 × 1.5 / 1 = 1.125

Option 2 wins despite protecting a less critical asset,
because its perimeter position (hᵈ = 1) more than
compensates for the lower asset criticality.

This is the perimeter-preference principle encoded in math.
A good honeypot at the front door beats a great honeypot
at the vault — because the front-door honeypot catches
the attacker before they reach the vault at all.

σⱼ ∈ [0, 1] is the stealth score for ATT&CK technique tⱼ. It measures how invisible this technique is to conventional security tools — firewalls, IDS, SIEM rules, endpoint detection. A score near 1.0 means the technique is nearly impossible to catch without a honeypot. A score near 0 means the technique is already well-covered by existing tools and a honeypot adds marginal value.

How σ enters the weight formula:


W_{j,a} = w̃_{j,a} × (1 + σⱼ)        (equation 3)
The (1 + σⱼ) factor scales the weight upward for stealthy techniques. Let us compute the full weight for two techniques:


T1048 (exfiltration, σ = 0.90) on oracle-prod (dₘ = 3.5, hᵈ = 3):
  w̃ = w_{T1048} × dₘ / hᵈ = 0.90 × 3.5 / 3 = 1.05
  W = w̃ × (1 + σ) = 1.05 × (1 + 0.90) = 1.05 × 1.90 = 1.995

T1046 (network scan, σ = 0.15) on oracle-prod (same asset):
  w̃ = w_{T1046} × dₘ / hᵈ = 0.40 × 3.5 / 3 = 0.467
  W = w̃ × (1 + σ) = 0.467 × (1 + 0.15) = 0.467 × 1.15 = 0.537
The db_trap covering T1048 on oracle-prod earns W = 1.995 per detection credit. The generic_trap covering T1046 on the same asset earns only W = 0.537. That is a 3.7× difference driven entirely by the combination of base weight and stealth score.

====

How w, dₘ, hᵈ, σ Combine: The Complete Pricing Formula
The three equations work as a pipeline:


Step 1 — Topology adjustment (equation 2):
  w̃_{j,a} = w_j × dₘ,a / hᵈ,a

  Takes the raw technique value (w_j),
  scales up for critical assets (× dₘ,a),
  scales down for deep assets (÷ hᵈ,a).

Step 2 — Stealth adjustment (equation 3):
  W_{j,a} = w̃_{j,a} × (1 + σⱼ)

  Takes the topology-adjusted value,
  scales up for techniques that existing tools miss.

Step 3 — Path weight (equation 4):
  PW_{π,h,j,a} = ρπ × iv_{π,h} × W_{j,a}

  Combines the fully adjusted technique-asset value
  with the path probability (from ρ) and
  the hop interception value (from iv).
  This is the credit the solver earns for one detection.
